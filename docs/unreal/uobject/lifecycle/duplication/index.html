对象拷贝

进阶内容
引擎版本：5.4更新日期：2024-09-27状态：In Progress
前置知识：对象实例化

UObject 内置对原型模式的支持，因此可以从任意 UObject 实例派生出一个新的 UObject 实例。 从类型生成实例的过程称为实例化，而从实例生成实例的过程就称为拷贝。 对象拷贝在一些情况下是非常有用的，比如启动 PIE 时就会拷贝一份当前地图的副本，让游戏运行在副本上，从而避免地图资源被错误地写入运行时数据。 又比如对象可以跨类型进行拷贝，可以仅以蓝图子类作为数据配置，运行时则生成一个固定父类的实例，从而提高实例的复用性。

对象拷贝的入口有多个，都包含 DuplicateObject 字样，它们最终都会转发给 StaticDuplicateObjectEx()。 这个函数主要做三件事：123、123、123。本文将围绕这个函数展开详细分析。

草稿

  1. 直接使用 Template 创建和 Duplicate 都是某种意义上的对象拷贝，但从结果上看，它们的行为有很多不同。
    1. 两者会对满足以下任意条件的子对象进行深拷贝：
      1. Default Subobject (DSO)，FObjectInitializer::CreateDefaultSubobject() 会直接去 Archetype 里找同名的 DSO。
      2. 被 Instanced 属性（属性上标记 Instanced 或属性的声明类型对应的 UClass 上有 DefaultToInstanced）引用的任何子对象，UStruct::InstanceSubobjectTemplates() 会扫描这些属性并拷贝。 注意这些属性指向的必须是当前对象的子对象，否则是浅拷贝。
    2. Duplicate 除了会深拷贝满足以上条件的子对象，还会拷贝被非 Instanced 属性引用的子对象，即 Duplicate 只要发现某个属性引用了子对象，就会进行深拷贝。
    3. 两者在 Instanced 属性的处理上有差异：
      1. Duplicate 会正确恢复所有属性的引用，因为 FDuplicateDataWriter 在写入 UObject 时实际写入的是去 DuplicatedObjectAnnotation 查出来的拷贝后的实例，因此 FDuplicateDataReader 恢复的时候能读到正确的值。
      2. 而 Template 在恢复 DSO 属性时故意跳过了非 Instanced 属性，导致非 Instanced DSO 属性会错误地指向 Template 的 DSO…… 不显式指定 Template 创建对象时，Template 就是 CDO，这会导致属性初始化阶段使用 PostConstructLink 而不是 PropertyLink，从而避免了非 Instanced DSO 属性的值被错误覆盖。
    4. Template 是直接在属性初始化阶段拷贝默认值，同时构建 DSO；而 Duplicate 在属性初始化阶段固定使用 DestClass 的 CDO 先拷贝一次默认值，但不构建 DSO，而是在外面通过序列化 (FDuplicateDataReader/Writer) 来从原型上拷贝默认值并构建 DSO。
    5. Duplicate 使用结构类型，只要 Source 的结构和目标类型相同即可拷贝，并且属性名不需要相同，但这里是非类型安全的，即没有运行时类型检查，需要开发者保证两个类型的兼容性；而 Template 使用标称类型，Source 的类型必须是目标类型或其子类才能拷贝。 即：Duplicate 可以跨类型拷贝，但开发者要保证两个类型结构完全一致；Template 可以从子类实例切片拷贝到父类实例，但不能在两个无关类型实例之间拷贝。

匿名 DSO 拷贝问题

偶然发现 ActorComponent 在作为 Actor 的 DSO 时无论是否有属性引用，都能正确拷贝；而普通 DSO 或者不挂在 Actor 上的 ActorComponent 则不会拷贝。

最先调试发现差异：ActorComponent 在作为 Actor 的 DSO 时，Archetype 是 Outer Archetype 的 DSO；而其他情况必须有属性引用，否则 Archetype 是 DSO 类的 CDO。 这将导致 IsDefaultSubobject() 判断异常，错误地认为它不是 Archetype 的 DSO，因此不会作为 DSO 的原型，也就不会拷贝。

又调试很久才发现真相：ActorComponent 与 Actor 成对使用时会防止 GC，如果 Outer 是 Actor，则 ActorComponent::PostInitProperties 会添加到 Actor::OwnedComponents 里， 而 OwnedComponents 虽然不是 UProperty，却通过 ARO 手动注册给 GC 了，所以 OwnedComponents 里的组件都不会被 GC，可以正常拷贝。 而对于普通 DSO，或者不挂在 Actor 上的 ActorComponent（极少见但有），就没有这个保护机制，所以 CDO 的 DSO 很快就会被 GC，于是在拷贝的时候直接就找不到原型了，所以也就不会拷贝了。

这与我们最早发现的差异也相吻合：有没有属性引用决定了会不会被 GC。只要 CDO 的 DSO 没被 GC，就是全部都会被正确拷贝的，而被 GC 的就无法正确拷贝了（没错，CDO 坏了，Template 和 Duplicate 也都会跟着坏，很难解释……）。

由此得出推论：DSO 一定要有引用，无论是自动引用（UProperty）还是手动引用（ARO）都可以，否则会导致拷贝功能异常。

遗留问题：调试过程中发现被 GC 的 DSO 正好被另一个 UObject 覆盖了，首地址完全对应上了一点都没偏，怀疑 UE 的内存分配器是分块的？
